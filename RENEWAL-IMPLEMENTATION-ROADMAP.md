# 기획안 2 구현 순서

> 기준 문서: `RENEWAL-PLAN-2-IDLE-RPG.md`
> 작업 위치: `card-gacha-renewal` 브랜치와 별도 worktree
> 운영 원칙: 사용자 지시 전까지 로컬 작업만 진행하고 커밋, push와 배포를 하지 않는다.

## 1. 작업 원칙

- 전체 기능을 동시에 만들지 않는다.
- 샘플 카드 20장, 지역 1개, 스테이지 10개로 첫 플레이 가능 버전을 만든다.
- 전투 계산과 저장이 검증된 뒤 카드 212장으로 확장한다.
- 월드보스는 모험 전투 계산기를 재사용할 수 있을 때 시작한다.
- 아이템 보급팩은 행동력, 카드 경험치와 강화 기능이 먼저 완성된 뒤 연결한다.
- 기존 `index.html`은 최종 전환 전까지 보존한다.
- 시즌2 화면은 `index.html`, 분리 CSS와 네이티브 ES 모듈로 개발한다.
- Supabase 마이그레이션은 기능 단계별로 나누고 되돌릴 수 있게 작성한다.
- 실제 계정·DB 작업은 `docs/RENEWAL-PRE-DATABASE-WORKLIST.md`의 `PDB-9` 승인 뒤 시작한다.

## 3. 단계 0: 수치와 데이터 규칙 잠금

구현 전에 다음 값을 코드가 읽는 단일 설정 파일로 확정한다.

- F~SSS 등급별 기본 능력치 배율
- 공격력, 체력, 방어력, 공격속도, 치명타 확률과 치명타 피해 계산식
- 강화 단계별 능력치 배율
- 단계별 카드 경험치 요구량
- 계정 레벨 없이 카드 등급·강화·도감·종족 시너지로만 성장하는 전투 공식
- 도감 보너스 구간과 전투 보너스 상한 100%
- 모험 일반 적, 정예 적과 보스 능력치 증가식
- 오프라인 보상 계산식
- 행동력 최대치, 회복 속도와 사용처
- 카드팩과 작전 지원 보급팩 확률

첫 데이터는 전체 카드가 아니라 샘플 20장으로 만든다.

- 등급별 카드 최소 2장
- 속공, 강타, 연타, 광역, 보스, 증폭, 약화와 생존 유형 포함
- 저그, 테란과 프로토스 포함
- 실제 상·중·하위 등급 분포 포함

완료 조건:

- 같은 입력은 항상 같은 전투 결과를 만든다.
- 모든 확률표 합계가 정확히 100%다.
- 전투력 계산식과 실제 전투 성능 차이가 과도하지 않다.
- 각 등급에 8개 유형이 모두 있고 유형별 수량 차이가 1장 이하다.
- 0성 등급별 도달 단계와 저등급 9성·S 9성·SS 5성 엔드게임 경계를 회귀 테스트로 잠근다.

## 4. 단계 1: 리뉴얼 앱 골격

기존 1,716줄 `index.html`에 새 게임을 계속 추가하지 않는다.

초기 파일 구조 후보:

```text
index.html
assets/renewal/
styles/renewal/
  tokens.css
  layout.css
  components.css
  responsive.css
src/renewal/
  app.js
  api.js
  state.js
  router.js
  screens/
  components/
  game/
```

먼저 만드는 화면:

- 공통 상단 바
- 좌측 메뉴
- 모험 중앙 전장
- 하단 카드 5장
- 우측 지역 진행 패널
- 모바일 하단 메뉴

이 단계에서는 가짜 샘플 데이터로 화면만 동작한다.

완료 조건:

- 승인된 이미지 시안과 전체 구성이 일치한다.
- PC와 모바일에서 카드 5장이 같은 비율로 표시된다.
- 메뉴 이동과 패널 전환 시 레이아웃이 흔들리지 않는다.
- 카드와 UI가 서로 겹치거나 화면 밖으로 넘지 않는다.

## 5. 단계 2: 전투 계산기

UI보다 먼저 순수 전투 로직을 만든다.

필수 모듈:

- 카드 최종 능력치 계산
- 공격 순서와 공격속도 처리
- 치명타 판정
- 카드 유형별 패시브
- 적 대상 선택
- 일반 웨이브와 보스 제한시간
- 승리, 실패와 피해 기록
- 전투 이벤트 로그 생성

서버는 난수 시드를 발급하고 결과를 계산한다. 브라우저는 서버 이벤트 로그를 애니메이션으로 재생한다.

완료 조건:

- 클라이언트가 피해량을 임의 수정할 수 없다.
- 전투 결과를 빠르게 계산할 수 있다.
- 전투 화면을 스킵해도 서버 결과가 동일하다.
- 자동 시뮬레이션으로 샘플 덱 수천 회 비교가 가능하다.

## 6. 단계 3: 로컬 계정과 편성 상태

로컬 세로 조각에서 편성·성장 상태와 저장 규칙을 먼저 검증한다. 실제 마이그레이션은 `PDB-9` 승인 뒤 작성한다.

테이블 후보:

- `game_profiles`: 전투력 캐시와 행동력
- `card_progress`: 카드별 강화 단계와 카드 경험치
- `formations`: 카드 5장 편성과 프리셋
- `adventure_state`: 현재 지역, 스테이지와 마지막 정산 시간

공개 버전에서는 기존 로그인 키와 사용자 ID를 그대로 사용하고 게임 데이터를 서버에 저장한다. 현재 로컬 저장은 화면과 게임 규칙 검증용이다.

완료 조건:

- 다른 기기에서 로그인해도 같은 편성과 진행도를 불러온다.
- 같은 카드를 한 편성에 중복 배치하지 못한다.
- EX와 미보유 카드를 편성하지 못한다.
- API 요청을 반복해도 중복 보상이 발생하지 않는다.

## 7. 단계 4: 모험 세로 조각

지역 1개와 스테이지 10개만 실제 구현한다.

구성:

- 일반 웨이브 9개
- 보스 스테이지 1개
- 샘플 적 3종
- 정예 적 1종
- 지역 보스 1종
- 편성 변경
- 자동 진행 ON/OFF
- 승리 후 다음 스테이지 이동
- 실패 후 현재 스테이지 반복

완료 조건:

- 로그인부터 편성, 전투, 승리와 다음 스테이지 이동까지 끊기지 않는다.
- 새로고침 후에도 현재 전투 상태가 복구된다.
- 화면을 닫아도 서버 진행 정보가 손상되지 않는다.
- 카드의 등급·강화 차이가 전투에서 확인된다.

이 시점이 첫 번째 내부 플레이 테스트다. 재미가 없으면 전체 카드와 콘텐츠를 만들기 전에 전투 속도와 연출을 수정한다.

## 8. 단계 5: 오프라인 보상

최고 도달 스테이지와 마지막 정산 시간을 기준으로 서버에서 계산한다.

- 최대 24시간 누적
- 계정 경험치
- 카드 경험치
- 성장 아이템
- 수령 전 예상량 표시
- 빠른 전투 2시간분 지급
- 중복 수령 방지

완료 조건:

- 브라우저 시간 변경으로 보상이 증가하지 않는다.
- 24시간을 초과해도 상한 이상 지급되지 않는다.
- 버프 시작·종료 시각을 나눠 정확히 계산한다.
- 편성 카드 5장에 카드 경험치가 정확히 지급된다.

## 9. 단계 6: 카드 경험치와 강화

기획안 1의 `9성` 강화 규칙을 가져오고 경험치 최대 조건을 추가한다.

구현 순서:

1. 카드 경험치 게이지
2. 강화 가능 상태 표시
3. 중복 카드 재료 선택
4. `1~3성` 확정 성공
5. `4~6성` 일반 실패
6. `7~9성` 파괴 판정
7. `9성` 카드뽑기 포인트 소모
8. 강화 촉진제와 파괴 차단제
9. 강화 성공 4단계 FX와 강화 구간별 강도 차등

완료 조건:

- 경험치가 부족하면 강화 API도 거절한다.
- 일반 실패 시 경험치는 유지되고 재료만 소모된다.
- 성공 시 경험치가 0으로 초기화된다.
- 파괴 시 원본 카드는 유지되고 강화 단계와 카드 EXP만 0으로 초기화된다.
- 강화 확률과 사용 아이템 효과가 화면 수치와 일치한다.
- 성공 연출은 충전·충돌·결과·정착 단계가 구분되고 데스크톱·모바일 가로에서 카드 정보와 겹치지 않는다.

## 10. 단계 7: 도감 계정 보너스

기존 카드 목록과 보유 수량을 하나의 도감 화면으로 통합한다.

- 보유·미보유 카드
- 인물, 종족과 등급 필터
- 카드 상세 능력치와 패시브
- 카드 잠금
- 인물·종족·등급·전체 등록률 컬렉션
- 계정 공격력·체력·방어력·보스 피해·방치 효율 보너스
- EX 전시 영역

완료 조건:

- 강화 재료로 카드를 사용해도 최초 획득 기록은 유지된다.
- 도감 전투 보너스 총합이 상한을 넘지 않는다.
- EX는 전투 보너스 계산에서 제외된다.
- 도감 변경 후 서버 전투력 캐시가 정확히 갱신된다.

## 11. 단계 8: 상점과 보급품

새 확률표를 서버 가챠 로직에 연결한다.

카드팩:

- 일반 카드팩
- 정예 카드팩
- 프리미엄 카드팩
- 저그·테란·프로토스 종족팩
- 확정 등급과 보장 슬롯 없음

작전 지원 보급팩:

- 전술 배터리 S/M/L
- 강화 촉진제
- 고순도 강화 촉진제
- 파괴 차단제
- 경험 신호 증폭제
- 고출력 경험 신호 증폭제
- 카드팩 교환권
- 10회 희귀 보급품 보장

추가 테이블 후보:

- `item_inventory`
- `gacha_transactions`
- `active_buffs`

완료 조건:

- 카드팩과 아이템팩의 확률 합계가 100%다.
- 10회 보장이 정확히 한 번만 적용된다.
- 포인트 차감과 지급이 하나의 트랜잭션으로 처리된다.
- 버프 중첩, 행동력 초과 충전과 강화 보조 제한이 기획과 일치한다.

## 12. 단계 9: 전체 카드와 모험 콘텐츠 확장

세로 조각 검증 후 전체 데이터를 만든다.

- 카드 212장의 기본 스탯
- 카드 유형 1개
- 패시브 1개
- 등급과 종족 데이터
- 지역별 배경
- 일반 적, 정예 적과 보스
- 스테이지 난도 곡선

카드 207개를 손으로 각각 코딩하지 않는다. 능력 템플릿을 만들고 카드 데이터가 템플릿 ID와 수치만 참조하게 한다.

완료 조건:

- 카드 데이터 스키마 검증 통과
- 패시브가 없는 카드 없음
- 특정 인물이나 종족만 모든 콘텐츠에서 우세하지 않음
- 낮은 등급도 일부 스테이지와 조합에서 사용할 이유가 있음

## 13. 단계 10: 미니게임

첫 공개는 `카드 짝맞추기`와 `캄몬사과게임` 두 게임으로 구현한다.

로컬 세로 조각 완료. Supabase 서버 검증과 실제 계정 포인트 트랜잭션은 공개 전 작업으로 남긴다.

공통:

- PC와 모바일 가로모드 입력
- 서버 시드와 입력 로그 검증
- 행동력 10을 사용하는 보상 모드
- 행동력 없는 무료 연습 모드
- 일일 포인트 상한
- 카드 전투력·강화·도감 보너스 미적용

카드 짝맞추기:

- 기본 4×4, 상급 6×6 보드
- 실제 카드 이미지의 동일 카드 2장을 짝으로 사용
- 제한시간, 시도 횟수, 연속 정답 기록
- 완전 클리어 보상 4×4 500P, 6×6 1,500P

캄몬사과게임:

- `Fruit Box`처럼 드래그 사각형 안 숫자 합이 10이면 타일 제거
- 120초 제한, 제거 타일 1개당 1점, 제거 후 재배치 없음
- `assets/card-back.jpg`를 격자로 잘라 타일 배경으로 사용
- 포인터·터치 드래그 좌표와 선택 결과를 서버에서 재현 검증
- 기본 40P와 제거 점수×1, 1회 최대 240P

완료 조건:

- 카드 전투력과 도감 보너스가 점수에 영향을 주지 않는다.
- 조작 자동화와 비정상 점수를 서버에서 거절한다.
- 정상 유저가 하루에 얻을 수 있는 포인트 범위가 카드팩 경제를 무너뜨리지 않는다.
- 카드백 타일이 제거되기 전에는 전체 이미지가 한 장처럼 정확히 이어진다.

## 14. 단계 11: 월드보스

모험 전투 계산기와 카드 편성을 재사용한다.

로컬 세로 조각 완료. 서버 공동 HP 원자 차감, 이벤트 스케줄, 중복 요청 방지와 실제 전체 랭킹은 Supabase 연결 단계로 남긴다.

추가 데이터:

- `world_boss_events`
- `world_boss_attempts`
- `world_boss_contributions`
- `world_boss_rewards`

구현 순서:

1. 출현 전 카운트다운
2. 이벤트당 3회 도전
3. 60초 전투 계산
4. 서버 공동 HP 차감
5. 개인 최고 피해와 누적 피해
6. 참여·피해 구간·처치 보상
7. 방송용 전체화면

완료 조건:

- 동시에 많은 유저가 공격해도 HP가 잘못 차감되지 않는다.
- 한 유저가 3회를 넘겨 공격할 수 없다.
- 동일 요청이 재전송돼도 피해가 중복 반영되지 않는다.
- 랭킹 갱신이 DB를 과도하게 읽지 않는다.

## 15. 단계 12: 전투력 랭킹

로컬 세로 조각 완료. 다른 콘텐츠 순위는 제외한다.

- 현재 편성 카드 등급, 강화, 종족 시너지와 도감 보너스를 합산한 전투력
- 상위 전투력 순위
- 50위 밖에서도 본인 전체 순위와 상위 백분위 표시
- TOP 50까지 필요한 전투력 차이

공개 버전은 정기 스냅숏을 사용하고 매 화면 요청마다 전체 사용자를 다시 정렬하지 않는다.

## 15.5. 계정·DB 진입 전 게이트

단계 13에 들어가기 전에 [계정 연동·데이터베이스 작업 전 체크리스트](docs/RENEWAL-PRE-DATABASE-WORKLIST.md)를 완료한다.

필수 순서:

1. UI·UX·브랜드와 필수 이미지 완성
2. 무안내 플레이테스트와 UX 버그 동결
3. 전투·성장·경제 밸런스 잠금
4. 게임 상태 스키마 v2 잠금과 v0·v1 자동 변환
5. UI와 저장 계층 분리
6. 서버 판정·API·멱등성 계약 확정
7. 로딩·오류·충돌 UI 구현
8. 성능·보안·모바일 사전 검수
9. 시즌 1 계정 이관 명세와 dry-run 규칙 확정
10. 사용자 DB 작업 시작 승인

`PDB-9` 전에는 리뉴얼 migration, RPC, RLS와 운영 데이터 이관을 실행하지 않는다.

## 16. 단계 13: DB, 기존 계정과 SOOP 이관

기존 사용자에게 다음 데이터를 연결한다.

- 로그인 키와 SOOP 계정
- 닉네임
- 카드뽑기 포인트
- 보유 카드와 수량
- 카드 시리얼
- SOOP 후원 포인트 지급

기존 등급은 전투에 사용하지 않고 `renewalRarity`를 사용한다. 단체사진 EX는 전투 편성에서 제외한다.

완료 조건:

- 운영 DB 복제본으로 이관 리허설 성공
- 사용자 수와 카드 총수 이관 전후 일치
- 포인트가 중복 지급되거나 사라지지 않음
- 기존 로그인 키로 새 게임 로그인 가능
