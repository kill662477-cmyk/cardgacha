# PDB-12 카드팩 구매·강화 원자 RPC

## 목적

포인트, 카드, 도감, 강화 재료와 보조 아이템을 클라이언트가 직접 판정하지 못하게 한다. 카드팩 구매와 강화 결과를 service role 전용 트랜잭션으로 확정한다.

## 파일

- SQL: `supabase/renewal_migration_004_pack_and_enhancement.sql`
- 정적 검증: `tests/renewal-pack-enhancement-rpc.test.js`
- 기반: `supabase/renewal_migration_001_accounts_reset.sql`부터 `003`까지

운영 Supabase에는 아직 실행하지 않았다.

## 추가 기록

### `gacha_s2_pack_draws`

팩 명령별 추첨 순서, 상품, 종족, 카드, 등급과 서버 시드를 보존한다. 같은 사용자·명령·추첨 순서는 중복 저장할 수 없다.

### `gacha_s2_enhancement_results`

강화 전후 단계, 성공·실패·파괴, 실제 확률과 roll, 재료, 보조제, 포인트와 서버 시드를 보존한다. 파괴 차단 기록은 파괴 차단제를 사용한 실패 결과에만 허용한다.

두 테이블 모두 RLS를 켜고 공개 직접 접근을 막는다. `gacha_s2_command_audit.server_seed`에도 성공 명령의 시드를 남긴다.

## 서버 난수

- `gacha_s2_new_seed`: `pgcrypto.gen_random_bytes(4)`로 명령 시드 생성
- `gacha_s2_seed_roll`: 시드와 카운터의 SHA-256으로 재현 가능한 `[0, 1)` roll 생성
- 호출자는 시드, roll, 등급, 카드, 강화 결과를 입력할 수 없음
- 시드는 명령 커밋 뒤 응답과 감사 로그에 공개해 결과를 재현할 수 있음

## `gacha_s2_purchase_pack`

입력은 사용자 UUID, 예상 revision, 멱등성 키, 상품 ID, 구매 수량 `1|10`, 종족팩 종족이다.

처리:

1. 사용자 상태 행을 `FOR UPDATE`로 잠근다.
2. 동일 멱등성 키는 최초 응답을 재생하고 변형 요청은 거부한다.
3. 활성 밸런스 JSON에서 가격, 장수와 등급 확률을 읽는다.
4. 서버 시드로 등급과 카드를 추첨한다. EX·단체사진은 제외하며 종족팩은 선택 종족만 허용한다.
5. 포인트 차감, 카드 수량 증가, 영구 도감 등록과 추첨 로그를 한 트랜잭션으로 처리한다.
6. revision과 상점 거래 횟수를 각각 1 증가시키고 응답·멱등성·감사 로그를 저장한다.

10회 구매는 팩 가격과 팩 장수를 각각 10배 적용한다. 일반 30장, 정예·프리미엄 40장, 종족 30장이다.

## `gacha_s2_enhance_card`

입력은 사용자 UUID, 예상 revision, 멱등성 키, 대상 카드, 다음 성급, 재료 카드 ID 배열과 선택 보조제다.

처리:

1. 사용자 상태와 대상 카드 행을 잠근다.
2. 대상 보유, EX 제외, 현재 성급 다음 단계와 카드 EXP 최대 조건을 검증한다.
3. 활성 밸런스의 등급별 재료 규칙을 적용한다. 잠금 카드와 마지막 1장은 재료로 쓸 수 없다.
4. 9성 포인트 비용과 보조제 보유량·사용 가능 단계를 검증한다.
5. 서버 시드로 성공·실패·파괴를 판정한다.
6. 결과와 무관하게 재료·선택 보조제·9성 포인트를 소모한다.
7. 성공은 다음 성급·EXP 0, 일반 실패는 현재 성급·EXP 유지, 파괴는 원본 카드 수량을 유지하고 성급·EXP만 0으로 초기화한다.
8. revision과 강화 시도 횟수를 증가시키고 결과·멱등성·감사 로그를 저장한다.

파괴 차단제는 파괴 판정을 실패로 바꾸며 시도 비용은 돌려주지 않는다.

## 보안 경계

- 두 RPC는 `service_role`만 실행 가능
- `anon`, `authenticated`와 `public` 직접 실행·테이블 접근 금지
- API 서버는 로그인 세션에서 사용자 UUID를 결정해야 함. 브라우저가 보낸 임의 UUID를 그대로 전달하면 안 됨
- 클라이언트는 성공 결과를 받은 뒤에만 카드팩·강화 FX를 재생

## 검증 제한

현재 환경에는 PostgreSQL/Supabase 로컬 런타임이 없어 정적 회귀만 완료했다. 실제 적용 전 staging에서 migrations `001→002→003→004` 순서 실행, RLS, 동시 요청, rollback과 확률 재현을 통합 검증해야 한다.
